# Milestone v1.0: Brownfield Hardening Baseline

**Status:** SHIPPED 2026-03-26
**Phases:** 1-4
**Total Plans:** 6

## Overview

This milestone hardened an already-working social security spreadsheet aggregation system without reopening its core business scope. The work focused on non-local auth safety, safe upload intake, reproducible dual-template export verification, and one supported operations path for operators and future agents.

## Phases

### Phase 1: Deployment Security Guardrails

**Goal**: Operators can only run authenticated non-local environments with explicit, non-default secrets and passwords.
**Depends on**: Nothing
**Plans**: 1 plan

Plans:

- [x] 01-01-PLAN.md - Enforce explicit non-local auth secret and password startup guardrails with regression tests.

**Details:**
- Added explicit runtime-environment handling and startup guardrails that reject default admin, HR, and auth-secret values outside local development.
- Kept local development bootable and preserved the supported auth API flow with safe credentials.

### Phase 2: Safe Upload Intake

**Goal**: File ingestion fails safely under oversized or invalid uploads without regressing the working import-to-export pipeline.
**Depends on**: Phase 1
**Plans**: 1 plan

Plans:

- [x] 02-01-PLAN.md - Enforce shared streamed upload limits and extend import/aggregate regression coverage without changing the existing pipeline contract.

**Details:**
- Added authoritative stream-time upload size enforcement and deterministic cleanup for partial files and failed batches.
- Surfaced oversized upload failures as explicit API errors while preserving the existing import and aggregate paths.

### Phase 3: Reproducible Export Verification

**Goal**: Dual-template export confidence is reproducible from repo-controlled or explicitly configured inputs instead of one developer workstation.
**Depends on**: Phase 2
**Plans**: 2 plans

Plans:

- [x] 03-01-PLAN.md - Establish repo-controlled export fixtures, shared export test helpers, and fail-loud explicit template resolution.
- [x] 03-02-PLAN.md - Migrate exporter, aggregate, and dashboard verification onto the shared fixture contract and remove missing placeholder dependencies.

**Details:**
- Added repo-controlled regression templates and a shared export fixture contract for tests.
- Removed Desktop-only lookup assumptions and made export-related regression coverage rerunnable on another machine.

### Phase 4: Supported Operations Path

**Goal**: Operators and future agents can follow one supported local workflow and one supported deployment workflow without confusing rescue tooling for the canonical path.
**Depends on**: Phase 3
**Plans**: 2 plans

Plans:

- [x] 04-01-PLAN.md - Define the canonical supported local and Linux/systemd deployment workflows in operator-facing docs without changing runtime behavior.
- [x] 04-02-PLAN.md - Classify rescue and legacy operational tooling, demote server-specific notes, and add explicit `.planning/` future-agent handoff guidance.

**Details:**
- Published one supported local path and one supported Linux/systemd deployment path.
- Moved rescue tooling under `scripts/operations/rescue/` and made `.planning/` the in-repo handoff surface.

## Milestone Summary

**Key Decisions:**

- Treat the repo as a brownfield baseline and focus v1.0 on hardening, verification, and operational clarity.
- Keep rules-first parsing, dual-template export, and provenance retention as fixed project constraints.
- Archive v1.0 with known audit gaps rather than reopen finished implementation work during closeout.

**Issues Resolved:**

- Unsafe default auth settings could no longer start in non-local environments when the guardrail is correctly configured.
- Streamed uploads could no longer bypass size limits through missing or incorrect `content-length` on the covered paths.
- Export verification no longer depends on one developer desktop path.
- Operators and future agents now have one supported path instead of a mix of rescue scripts and server-specific notes.

**Issues Deferred:**

- Fix the canonical deployment env contract so it actually activates Phase 1 guardrails.
- Fix the supported quick-aggregate `/aggregate/stream` path so streamed upload enforcement happens on the primary UI entrypoint.
- Backfill formal verification artifacts for Phase 1 and Phase 2.

**Technical Debt Incurred:**

- v1.0 closed with a failing milestone audit.
- Phase 3 Windows reruns remain timeout-prone.
