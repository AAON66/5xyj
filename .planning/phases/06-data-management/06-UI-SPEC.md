## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, pill internal padding |
| sm | 8px | Compact element spacing, button gaps |
| md | 16px | Default element spacing, form field gaps |
| lg | 24px | Panel card padding (`.panel-card` uses `24px`) |
| xl | 28px | Page content padding (`.app-content { padding: 28px }`, confirmed in styles.css line 291) |
| 2xl | 30px | Page hero padding (`.page-hero { padding: 30px }`, confirmed in styles.css line 304) |
| 3xl | 48px | Major section breaks |

**Source:** Measured from existing `.app-content { padding: 28px }`, `.panel-card { padding: 24px }`, `.page-hero { padding: 30px }`, `.status-item { padding: 15px 18px }` in styles.css.

**Non-standard value notes:**
- `xl` = 28px — inherited value from `.app-content`. Not a standard multiple of 4 but is a locked existing value; do not use 28px for any new elements introduced in this phase.
- `2xl` = 30px — inherited value from `.page-hero`. Not a multiple of 4 and not 32px; this is the actual measured CSS value. Locked; do not use 30px for any new elements introduced in this phase.

Exceptions:
- Nav link padding: 14px 16px — legacy value inherited from existing nav design; 14px is not a multiple of 4 but is locked from Phase 1 nav implementation. Do not apply to any new elements introduced in this phase.
- Button min-height: 44px (touch target floor — all `.button` elements, source: styles.css line 371). Declared as intentional UX exception, not a spacing token.
- Sidebar inner gap: 28px — same locked inherited value as `xl`; applies to existing sidebar only, do not reuse.
- Tab switching controls: 8px gap between tab labels (standard `sm` token, no exception needed).
- Secondary body text at 14.7px (0.92rem) sits near the 16px body baseline — this proximity is an inherited constraint from the existing nav design, not a new decision. Do not introduce additional sizes in this 14-16px range.

## Page Structure: 数据管理 (DataManagement page)

### Route
`/data-management`

### Layout
```
PageContainer
  eyebrow="Data Management"
  title="社保数据管理"
  description="按地区、公司和月份筛选全量社保明细，查看全员汇总数据。"
  actions=[ 无主动作按钮 ]

  [FilterBar]             ← 联动筛选器（地区 → 公司 → 月份）
  [TabBar]                ← 「明细数据」 | 「全员汇总」
  [TabContent]            ← 根据 activeTab 渲染不同表格
```

### FilterBar
Three cascading `<select>` elements in a `.employee-toolbar`-style flex row:
- 地区 — loads from API on mount, resets 公司 and 月份 on change
- 公司 — filtered by selected region, loads from API when region changes
- 月份 — shows available billing periods as YYYYMM formatted to "YYYY年MM月"
- A `.button--ghost` 「重置筛选」button that clears all three selects
- Filter state syncs to URL query params: `?region=shenzhen&company=xxx&period=202602`
- Filter change resets pagination to page 0 (source: D-04 from CONTEXT.md)

### 明细数据 Tab (DATA-01)

Table columns (default visible):
| Column | Width hint | Notes |
|--------|-----------|-------|
| 姓名 | 80px | person_name |
| 工号 | 80px | employee_id |
| 地区 | 70px | region label (Chinese) |
| 公司 | 120px | company_name |
| 月份 | 80px | billing_period as YYYY年MM月 |
| 单位合计 | 90px | company_total_amount, right-align, 2 decimal |
| 个人合计 | 90px | personal_total_amount, right-align, 2 decimal |
| 总额 | 90px | total_amount, right-align, 2 decimal |
| 操作 | 60px | 「展开详情」button |

Row expand interaction (D-03 from CONTEXT.md):
- Clicking 「展开详情」 toggles an inline sub-row below the row; button label changes to 「收起详情」
- Sub-row spans full table width via `colspan`
- Sub-row shows a two-column grid: 单位各险种 | 个人各险种
- Each insurance type shows: label + amount (right-aligned)
- Sub-row background: `rgba(243, 246, 251, 0.96)` (matches `.status-item`)
- Only one row expands at a time (expanding another collapses the previous)

Pagination:
- Same pattern as Employees.tsx: prev/next buttons + page indicator
- Default page size: 20 rows
- Page size options: 10, 20, 50

### 全员汇总 Tab (DATA-02)

Tab-level dimension switcher (`.tab-bar` pattern) with:
- 「按员工汇总」 — one row per employee, columns: 工号 | 姓名 | 公司 | 地区 | 最新月份 | 单位合计 | 个人合计 | 总额
- 「按月份汇总」 — one row per billing period, columns: 月份 | 总人数 | 单位合计 | 个人合计 | 总额 | 平均个人 | 平均单位

Both sub-views use the same filter state from FilterBar. Amounts right-aligned. No row expand in summary view.

## Dashboard Enhancement (DATA-03)

### New section added to existing Dashboard.tsx

Position: after the existing pipeline/status distribution cards, before the recent batches section.

Section heading: `panel-label` = "DATA QUALITY", h2 = "导入质量监控"

Layout: `.dashboard-grid` (3 columns) with 3 metric cards using `.panel-card`.

Three metric cards:
1. 缺失字段 — count of records missing 姓名 or 身份证号 or 工号 (`.dashboard-pill--warn` if > 0)
2. 异常金额 — count of records with payment_base outside valid range (`.dashboard-pill--warn` if > 0, `.dashboard-pill--danger` if > 10)
3. 重复记录 — count of duplicate person+period combinations (`.dashboard-pill--danger` if > 0)

Each card structure:
```
.panel-card
  .panel-label  — metric category name
  strong        — count number (large, 1.25rem weight 700)
  div           — label ("条缺失字段记录")
  .dashboard-pill--{tone}  — severity indicator
```

Per-batch breakdown:
Below the 3-card grid, add a `.status-list` showing each recent import batch with its quality counts. Each item is a `.status-item` row: batch filename | 缺失 N | 异常 N | 重复 N.

Loading state: `<SectionState title="正在加载质量指标" message="系统正在统计各批次数据质量情况。" />`
Empty state: `<SectionState title="暂无质量数据" message="尚未导入任何数据，质量指标将在首次导入后显示。" />`

## Interaction States

Every async operation must show one of these states. No silent loading.

| Operation | Loading | Success | Error |
|-----------|---------|---------|-------|
| Initial page load | `SectionState` with loading title | Table renders | `SectionState tone="error"` |
| Filter change | Table body shows `SectionState` (inline, not full page) | Table re-renders | `SurfaceNotice tone="error"` |
| Tab switch | Instant (data already loaded or triggers new fetch with `SectionState`) | Table renders | `SectionState tone="error"` |
| Row expand | Instant (data is in-row, no API call) | Sub-row appears | not applicable |
| Dashboard quality load | `SectionState` within quality section only | Cards render | `SectionState tone="error"` within quality section |
| Pagination | Table body `SectionState` | Table re-renders | `SurfaceNotice tone="error"` |

## URL Persistence Contract (D-04)

Filter state is reflected in URL query params. Rules:

- `region` — string, omit if empty string
- `company` — string, omit if empty string
- `period` — string (YYYYMM format), omit if empty string
- `tab` — "detail" or "summary", default "detail", always in URL
- `page` — integer starting at 0, omit if 0
- `page_size` — integer, omit if 20 (default)

On mount: read from URL, populate filter state, trigger initial fetch.
On filter change: push new URL, reset page to 0.
On tab change: push new URL, preserve filter params.
