## Cross-Phase Findings

### 1. Canonical deployment path does not activate Phase 1 guardrails correctly

- Severity: High
- Affected requirements: `SEC-01`, `SEC-02`, `OPS-01`
- Evidence:
  - Supported docs tell operators to set `ADMIN_PASSWORD` and `HR_PASSWORD`
  - The backend guardrails read `admin_login_password` and `hr_login_password`
  - `runtime_environment` still defaults to `local`, so the non-local guardrail is not guaranteed to turn on through the documented path

### 2. Primary quick-aggregate UI path buffers uploads before Phase 2 stream-time enforcement runs

- Severity: High
- Affected requirements: `PIPE-01`, `PIPE-02`
- Evidence:
  - The supported frontend flow goes through `/api/v1/aggregate/stream`
  - That endpoint reads the full upload into memory before re-wrapping it for the aggregate pipeline
  - The authoritative Phase 2 limit check still lives in `_store_upload()`
  - There is no oversized `/aggregate/stream` regression to prove the supported UI path meets the streamed limit guarantee

## Residual Risks

- Phase 3 fixture wiring and export verification are healthy, but the deployment and quick-aggregate entrypaths still need correction before a future audit can pass.
- Combined Windows pytest reruns remain slow enough to time out in this environment.
- Phase 1 and Phase 2 still lack formal phase-level verification reports.
